Funmands also introduces the concept of handling the pre-execution
of a command. The pre-execution of a command handles many things,
including:
- Set a predicate for the future sender of the command
  (can they see and execute it?)
- Dynamic runtime suggestions for commands
```java
pre.withSuggestions("bar", sender -> {
        return List.of(
        new Suggestion("foo", Component.text("my amazing tooltip!")),
        new Suggestion("bar", Component.text("so declarative!")));
        });
```
![The suggestions from the code above and their tooltips](docs/images/suggestions.png)

Those, however, are not the main selling points of the framework.
At the end of the day, it is just a **wrapper** around Brigadier
via a DSL. Humans are prone to errors, and so, a DSL done
incorrectly is **worse** than code that will never make a mistake.
However, that is not the case here. Funmands comes with not only
an IntelliJ plugin that shows the names of arguments, their types,
syntax errors, and any other warnings or errors that may
arise from the DSL format, but also a VSCode extension.
Below are the features of the plugin.

Hovering over a format will show you
the entire tree generated by Brigadier; there aren't any secrets.
The window containing the tree can be moved around and resized
to your liking. Not only will the entire tree be visible,
but your whole command structure will be as well. You will see
a view like this:
```java
LiteralArgumentBuilder.literal("libexample")
    .then(LiteralArgumentBuilder.literal("such")
    .then(LiteralArgumentBuilder.literal("cool")
    .then(LiteralArgumentBuilder.literal("float")
    .then(RequiredArgumentBuilder.argument("float", FloatArgumentType.floatArg())
        .then(LiteralArgumentBuilder.literal("int")
    .then(RequiredArgumentBuilder.argument("int", IntegerArgumentType.integer())
        .then(LiteralArgumentBuilder.literal("long")
    .then(RequiredArgumentBuilder.argument("long", LongArgumentType.longArg()).executes(context -> {
        // do command logic here
        })))))))));
```
```
/libexample such cool float <float> int <int> long <long>
```
where inside of the ``<>`` are the types that the user will input.
Hovering over the root of the command, you will see all possible formats,
along with all brigadier trees. You can collapse the formats and trees
as you wish.

In edge cases where you need direct access to brigader for some reason,
you can get a wrapped brigadier tree from a format via:
```java
this.asTree("some format <int:amount>");
```
upon which, you can treat it is a regular brigadier tree with the
benefits of the wrapper. You can use
```java
tree.get("some");
```
to go to the part of the brigadier tree where ``some`` was constructed, and
inject arguments after ``some``.
In order to go back to the DSL, you can simply do:
```java
String dsl = tree.toDsl();
```

Now, playing Minecraft and going into it is time-consuming
when testing commands. Why do that when you can test
them directly from your IDE? Commands can be simulated by
right clicking on a format and clicking ``Simulate Command``.
This will open a window in which, if the format has arguments,
you will input them and then simulate the command
as if you were doing it in game. This works with all
data types, including custom user-defined ones. Any text
components that would be sent to command senders or
executors will be serialized into strings and sent straight
into this window, in the ``Log`` panel. Tab completion is just as
important as commands, and so, you can see tab completions
straight from the window. If a tab completion is overridden
and relies on values at runtime, you can add your own
fake suggestion provider in order to get all the benefits
of tab completion. In the case that you would like to
test predicates, you can add permissions to a fake
command sender, along with choosing whether they are
operator or not. You can also save profiles for command testing,
with a profile saving both the arguments (if any) and the
predicate. Lastly, you can keep track of all simulations ran
in the current project in the history view of the window.
It will show you all ran formats, their profile, and their output.
If using the ``asTree()`` method on a format, you will be able
to view the constructed tree with all injections by hovering
over it.

Using ``Ctrl + Click`` on a data type (e.g. ```"int:cool"```)
will take you to the class definition of the type. Errors
can be caught ahead of time via static DSL analysis of the command
format. For example, if you have the format ``string_greedy:something``
and use ``args.get("somethng")``, an error will show up underneath
that line letting you know that the argument type name
does not exist. The quick fixes (which can be shown by hovering
over it) will ask you if you ``Argument name: "somethng" does not exist.
Did you mean "something"?`` Clicking on the fix will change the string
for you. If you declare an argument but never use it in code,
the plugin will give you a warning. If a format is defined twice,
you will also receive a warning.

Whilst typing a format, all known data types will have
auto-complete available, similar to how IntelliJ handles things like
variables.

Of course, you may be wondering how to actually **use** Funmands.
Funmands has first-class support for Paper; to register a command:
```java
final PaperCommandManager commandManager = new PaperCommandManager(this.getLifecycleManager());
commandManager.register(new LibExampleCommand());
commandManager.register(new SomeCommand(), new CommandMeta("Do something.", "stuff", "nice")); // The meta allows for descriptions & aliases.
```
Additionally, you can hook into the registration of a command via:
```java
commandManager.onRegistered(command -> logger.info("Command: " + command.getName() + "registered!"));
```
